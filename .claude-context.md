# Contexto del Proyecto Github-Smithy (Mini-GitHub Smithy)

## Resumen del Proyecto
Repositorio **contract-first** que define la API REST del ecosistema **Mini-GitHub** en **Smithy 2.0** (namespace `com.minigithub`). No es un microservicio en ejecución: es el **modelo canónico** de operaciones, rutas HTTP, tipos y autenticación Bearer; a partir de él se valida el modelo y se genera **OpenAPI (Swagger)** para implementaciones (Node, Java, etc.) y documentación.

**Servicio agregado en Smithy:** `com.minigithub#MiniGitHubApi`  
**Proyección OpenAPI:** `restJson1` → `build/smithyprojections/mini-github-smithy/openapi/openapi/MiniGitHubApi.openapi.json`

## Relación con los otros contextos

### [github-docs](https://github.com/dtarqui/github-docs) (`C:\Python\github-docs`)
- **github-docs** describe la visión de producto: microservicios (Auth, Repo, Issue, Search), gateway, eventos RabbitMQ, entidades y requisitos (HUs, RF).
- **Github-Smithy** materializa esa visión en un **contrato único versionable**: mismos dominios (auth, repo, files, issue, search) y puertos lógicos (3001–3004) reflejados como operaciones del servicio `MiniGitHubApi`.
- La documentación en `github-docs\docs\` sigue siendo la fuente de **historias de usuario y límites**; Smithy es la fuente de **forma exacta de la API** (paths, métodos, esquemas) para que implementaciones no diverjan.

### [Github-Cdk](https://github.com/dtarqui/Github-Cdk) (`C:\Python\Github-Cdk`)
- **Github-Cdk** provisiona **infraestructura AWS** (VPC, EKS, RDS, Keycloak en K8s) donde corren las aplicaciones.
- **Github-Smithy** no despliega nada: define **qué** exponen las APIs que eventualmente corren detrás del gateway o en el cluster. El CDK y Smithy se complementan: **IaC + contrato de API**.

### [Github-ms-users](https://github.com/dtarqui/Github-ms-users) (`C:\Python\Github-ms-users`)
- **ms-usuarios** es un microservicio **Spring Boot** centrado en usuarios corporativos, Keycloak Admin API, Eureka, doble PostgreSQL, RabbitMQ (`usuario.creado`, etc.).
- El **Auth Service** del Mini-GitHub en Smithy (registro, login, JWT, OAuth, perfil) es el contrato del **clon académico** descrito en github-docs; puede **inspirarse o alinearse** con patrones de ms-usuarios (JWT, Keycloak), pero **no es el mismo código ni el mismo dominio** (Mini-GitHub vs Inspire/SGSI).
- OpenAPI generado desde Smithy puede usarse en **springdoc-openapi** u otros generadores solo como **referencia de contrato**; la implementación real de Auth puede seguir siendo Node (docs) o coexistir con servicios Java según arquitectura elegida.

## Estructura del repositorio

```
Github-Smithy/
├── build.gradle                 # Plugin software.amazon.smithy, dependencias smithy-*
├── settings.gradle
├── smithy-build.json            # Fuentes model/, proyección openapi
├── setup.sh                     # Genera Gradle Wrapper
├── gradlew, gradlew.bat
├── model/
│   ├── service.smithy           # MiniGitHubApi: lista todas las operaciones
│   ├── common/common.smithy     # Tipos, enums, errores compartidos
│   ├── auth/auth.smithy         # Auth (HU-01 … HU-05)
│   ├── repo/repo.smithy         # Repos, branches, stars, forks, colaboradores
│   ├── files/files.smithy       # Contenido de archivos, commits, diff, carpetas
│   ├── issue/issue.smithy       # Issues, labels, comentarios, PRs
│   └── search/search.smithy     # Búsqueda repos / usuarios / issues
└── docs/
    └── file-management-spec.md  # Especificación detallada archivos/Git (Gitea, HUs)
```

## Dominios y operaciones (resumen)

| Área lógica | Puerto de referencia (docs) | Contenido en Smithy |
|-------------|-----------------------------|----------------------|
| Auth | 3001 | Register, Login, Logout, RefreshToken, GetMe, UpdateProfile, OAuth, Forgot/Reset password, GetUserByUsername |
| Repo | 3002 | CRUD repos, archive, branches, fork, star, colaboradores |
| Files | 3002 (mismo servicio repo) | Get/Create/Update/Delete file, folder, raw, commits, diff, compare |
| Issue + PR | 3003 | Issues, comentarios, labels, pull requests (review, merge) |
| Search | 3004 | SearchRepositories, SearchUsers, SearchIssues |

## Autenticación en el modelo
- Trait `@httpBearerAuth` a nivel de servicio.
- Operaciones protegidas: header `Authorization: Bearer <jwt>` (token típicamente vía `POST` login en prefijo documentado en shapes `auth`).

